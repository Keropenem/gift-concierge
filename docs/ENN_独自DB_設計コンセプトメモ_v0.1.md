## 1. 設計の根本思想

### 1.1 普通のEC商品DBを作らない

ENNの提案ロジック（Step 4〜6: Analysis → Narrative → Proposal）は、「二人の見えない共通点・背景の感情」と商品をマッチングする。
したがって独自DBに必要なのは、SKU・在庫・価格などのEC標準項目ではなく、**LLMが文脈マッチングに使える"物語側のメタデータ"** が中心となる。

**設計原則:**
- 物語側メタデータをファーストクラスで設計する
- EC標準項目（在庫・配送等）は副次的に持つ
- 「DBに登録されているから提案する」ではなく「文脈に合うから提案する」

### 1.2 独自DB優先の "正しい" 実現方法

独自DBを優先するが、**顧客に「自分たちの土俵に引き込まれた」と思わせてはいけない。**

→ 仕組みで強制するのではなく、**候補の渡し方**で実現する。
LLMに「独自DB候補 + Web検索候補」を並べて提示し、文脈整合性で最終選択させる。
独自DBの優先性は「最初に候補に入る」というだけで、合わなければWeb検索結果が選ばれる。

## 2. 商品メタデータ構造

### 2.1 標準項目（EC基本）

| 項目 | 備考 |
|---|---|
| 商品ID | 内部識別子。バリエーション（色・サイズ・素材違い等）が必要な商品はSKU (Stock Keeping Unit)単位で管理。一点物・受注生産は商品ID単位で完結 |
| 商品名 | |
| 価格 | バリエーション可 |
| 商品画像 | 複数枚 |
| 在庫状態 | 在庫あり／受注生産／一点物 等 |
| 配送情報 | 配送日数・地域制限 |
| 製造元・クリエイター情報 | 紐付け |
| 購入導線 | 購入経路（外部リンク／自社EC内決済 等の区分）と、対応するURLや内部商品ページ識別子。フェーズに応じて拡張可能な構造とする |

### 2.2 物語側メタデータ（独自DBの核）

| 項目 | 内容 | 用途 |
|---|---|---|
| 作り手のストーリー | 誰が・なぜ作ったか | Step 5 Narrativeの素材 |
| 込められた想い・哲学 | 制作者の意図 | Step 4 Analysisとの照合 |
| 想定贈答シーン | 誕生日／結婚／節目／お礼／感謝 等 | タグフィルタ |
| 想定関係性 | 親友／恋人／恩師／家族／同僚 等 | タグフィルタ |
| 喚起される感情キーワード | 懐かしさ／温かさ／挑戦／静謐 等 | ベクトル検索の意味軸 |
| 共鳴しやすい嗜好性タグ | 手仕事／自然／美意識／伝統 等 | タグフィルタ |
| 制作背景・素材・技法のナラティブ | 産地・工程・歴史 | Step 5 Narrative素材 |

## 3. メーカー／クリエイター登録フロー

### 3.1 設計方針

メーカーは「物語側メタデータ」を上手く文章化することに慣れていない。一方で対話型UIは鬱陶しがられる可能性がある。
→ **フレキシブルな入力方式**を提供し、登録者が自分に合った方法を選べるようにする。

### 3.2 入力方式（3パターン併用）

**(A) テンプレート直接入力**
- 標準項目・選択式タグは登録者が自分で入力
- 文章系（ストーリー・哲学・ナラティブ）の枠は空欄でも登録可能
- 「最低限の必須項目だけ埋めればOK」のラインを設定

**(B) LLM補助入力（対話 or 穴埋め）**
- 文章系の空欄をLLMが質問しながら埋める
- 「対話モード」と「穴埋め提案モード」を選択可
- 登録者が自分の言葉で語れる場合は対話、忙しい場合はLLM案を承認するだけ

**(C) 既存資料からの自動転記**
- メーカーが持っているパンフレット・ブランドブック・商品説明文をアップロード
- LLMが読み込んでテンプレートに自動転記
- 不足部分のみ (A)(B) で補完

### 3.3 キュレーション・審査

- **誰でも登録可ではない**：BEt社が関わったメーカー・クリエイターのみ
- 登録は招待制／申請制（運用詳細は別途）
- BEt社による品質確認プロセスを挟む

### 3.4 登録者ダッシュボード

- 提案露出回数・クリック数・購入導線への遷移数の可視化
- 商品情報の更新UI
- メーカー側の改善ヒントとなる情報の提供

## 4. 検索・マッチングロジック（最重要論点）

### 4.1 三段構えのハイブリッド検索

**結論: タグ → ベクトル → LLM の三段階で、独自DBとWeb検索を統合する。**

```
[Step 4 Analysis結果]
        ↓
[タグフィルタ層] 独自DB全件 → 数十件
  ・贈答シーン／関係性／価格帯／配送可否などの構造化条件で足切り
  ・"物語のこじつけ"には使わない、明らかにNGな候補を弾くだけ
        ↓
[ベクトル類似度層] 数十件 → 上位5〜10件
  ・Step 4のAnalysis結果と「ストーリー文・込められた想い」を埋め込み比較
  ・pgvector等で意味マッチング
        ↓
[Web検索並列実行] Geminiで補完候補を取得
        ↓
[LLM最終判定層]
  ・独自DB候補 + Web検索候補をLLMに提示
  ・Step 5 Narrativeとの整合性で最終選択
  ・独自DBが合わなければWeb検索結果を選ぶ権限をLLMに付与
```

### 4.2 各層の役割と理由

| 層 | 役割 | なぜこの設計か |
|---|---|---|
| タグ | 高速な粗い網 | 純粋ベクトルだと遅い／タグ単独だと"こじつけ"になる。明らかなミスマッチを弾くのみに留める |
| ベクトル | 意味マッチング | 物語ベースの照合はタグでは表現しきれない。pgvectorで実装可能 |
| LLM最終判定 | 物語の整合性 | 「DB優先」を仕組みでなく判断で実現。土俵に引き込む感を回避 |

### 4.3 独自DB / Web検索のハイブリッド方針

- 両者を**並列に**LLMへ渡す（独自DBが先に決まってからWeb検索する直列ではない）
- LLMには「独自DB候補は優先的に検討せよ。ただし文脈に合わなければWebを選んでよい」と指示
- 結果として、独自DB商品の採用率は自然と上がるが、無理に押し込まない
- **採用ログを蓄積**し、独自DBが選ばれなかったケースを分析 → メタデータ改善・登録商品拡充にフィードバック

### 4.4 レイテンシ見込み

- タグフィルタ：数十ms
- ベクトル検索：数百ms
- LLM最終判定：既存Step 6の処理に統合
- → 現MVPの30秒制約内に収まる見込み

## 5. 運用・データ品質（概略）

詳細は別途検討するが、最低限以下を要件に含める：

- 在庫・価格・URLの鮮度維持（メーカー側更新 + 自動チェック）
- リンク切れ・廃番検知
- 商品ライフサイクル（公開・一時停止・終売）
- 採用されなかった商品のメタデータ改善ループ
